## ADDED Requirements

### Requirement: Infrastructure-as-code Blueprint

The system SHALL define a Render Blueprint that provisions a Django web service, a static-site frontend, and a managed PostgreSQL database, so the full stack can be created from version control.

#### Scenario: Blueprint provisions all three resources

- **WHEN** the `render.yaml` Blueprint is applied on Render
- **THEN** it creates a Python web service running the Django app under gunicorn, a static site for the built frontend, and a PostgreSQL database whose connection string is supplied to the web service as `DATABASE_URL`

#### Scenario: Secret key is generated, not hardcoded

- **WHEN** the web service is provisioned
- **THEN** `DJANGO_SECRET_KEY` is generated by the platform and `DJANGO_DEBUG` is set to false

### Requirement: Reproducible build with populated database

The system SHALL provide a build step that installs production dependencies, collects static assets, applies migrations, and seeds reference and watch-party data idempotently.

#### Scenario: Build seeds the database

- **WHEN** the API build step runs during a deploy
- **THEN** it installs the PostgreSQL driver, runs `collectstatic`, applies migrations, and runs the idempotent reference-data and watch-party loaders so the database is populated

#### Scenario: Re-deploy is safe

- **WHEN** the build step runs again on a subsequent deploy
- **THEN** the seed loaders upsert by natural key without creating duplicates

### Requirement: Frontend points at the deployed API

The system SHALL build the static frontend with its API base URL derived from the deployed API service host.

#### Scenario: API base baked at build time

- **WHEN** the static site is built
- **THEN** `VITE_API_BASE` is composed from the API service's host and inlined into the production bundle so the SPA calls the deployed API

### Requirement: Production-hardened Django configuration

The system SHALL apply production security settings when `DEBUG` is false, and SHALL trust the platform-provided external hostname.

#### Scenario: HTTPS hardening in production

- **WHEN** the app runs with `DEBUG` false behind Render's TLS-terminating proxy
- **THEN** it enables SSL redirect, secure session and CSRF cookies, HSTS, and trusts the `X-Forwarded-Proto` header, and serves collected static files via WhiteNoise's compressed manifest storage

#### Scenario: Platform hostname is trusted

- **WHEN** the platform injects `RENDER_EXTERNAL_HOSTNAME`
- **THEN** it is added to `ALLOWED_HOSTS` and to `CSRF_TRUSTED_ORIGINS` so the default service URL serves requests and admin logins without hardcoding

### Requirement: Local development is unaffected by production settings

The system SHALL preserve the zero-setup local workflow: SQLite, the development server, and the test suite MUST work without running `collectstatic` or serving over HTTPS.

#### Scenario: Dev runs on SQLite with default storage

- **WHEN** the app runs with `DEBUG` true and no `DATABASE_URL`
- **THEN** it uses the local SQLite database and Django's default static storage, applies no HTTPS redirects or secure-cookie requirements, and the development server and full test suite pass without `collectstatic`

### Requirement: Custom-domain DNS helper

The system SHALL provide a scriptable way to create or update the Cloudflare CNAME that points the custom domain at the Render frontend, given a DNS-scoped API token and the Render target.

#### Scenario: Create or update the CNAME

- **WHEN** the DNS helper runs with a Cloudflare token (Zone:Read + DNS:Edit) and the Render CNAME target
- **THEN** it resolves the zone, then creates the custom-domain CNAME or updates it in place if it already exists, defaulting to DNS-only since Render manages the certificate
